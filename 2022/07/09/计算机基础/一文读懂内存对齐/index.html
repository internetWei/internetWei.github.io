一文读懂内存对齐
2022-07-09|计算机基础
字数总计: 6k|阅读时长: 22 分钟

结构体的大小和基本数据类型的大小以及外字节对齐大小都有关，所以想计算结构体的大小首先要确定外字节对齐大小和基本数据类型的大小。

通常情况，不同位数的环境下默认外字节对齐大小和基本数据类型的大小都不一样，例如在32位环境下long类型通常占4个字节，外字节对齐默认是4；而64位环境下long类型通常占8个字节，外字节对齐默认是8。即使同位数但不同的系统可能也不一样，例如在64位的Windows系统下long类型通常占4个字节，而64位的Mac OS系统下long类型却占8个字节(其它类Unix系统下的long类型也是占8个字节，例如Linux)。

文章中的所有代码都是在64位的Mac OS系统下运行的，系统版本是12.1，运行工具是Xcode，Xcode版本号为13.1，如果您是在64位的Windows系统或其他系统下运行示例代码，那么结果可能会不一样，请悉知。

经过手动测试，当前系统环境下外字节对齐默认值是8，基本数据类型大小如下：

类型 大小(字节)
char 1
short 2
int 4
long 8
float 4
double 8

要计算结构体的大小，首先要明白内对齐和外对齐。

内对齐

第1个数据成员的首地址始终是0，不需要进行内对齐；之后的每个数据成员内对齐规则是：使用外字节对齐大小和自身类型大小进行比较，取最小值，首地址必须是最小值的倍数。

外对齐

在所有数据成员完成内对齐之后，还需要对数据结构(结构体)本身进行对齐，外对齐的规则是：获取结构体中最大 基本数据类型(注意struct、union这些都不是基本数据类型)的值，和外字节对齐大小进行比较，取两者中的最小值；外对齐之后的大小必须是这个最小值的倍数。

  2. 计算外对齐时，最大的基本数据类型是有可能存在于内部其他结构体中的。

前面我们一直在讲结构体的对齐方式，其实共用体(union)也需要对齐。共用体不需要进行内对齐，只需要外对齐。

共用体的大小计算分为2步：

  1. 获取共用体内最大的那个变量大小，注意这里获取的是变量大小，不是最大的基本数据类型大小，如下共用体uu1最大的变量是s1，占用40个字节。
  2. 取内部最大的基本数据类型和外字节对齐进行比较，取两者中的最小值，最终大小必须是最小值的倍数；以uu1为例，其内部最大的基本数据类型是变量s1的变量b，double类型，和外字节对齐相比，最终大小应该是8的倍数，由于40是8的倍数，所以不需要补充字节进行对齐。

枚举的大小计算比共用体的大小计算更简单，一眼就能看出来；

枚举的总大小由内部最大的那个值决定，以e1为例，最大的值是2(blue)，int类型，所以e1的大小是int的大小4个字节。

以e2为例，最大的值是4294967295(orange)，这是 unsigned int 类型所能容纳的最大值，所以它的类型是 unsigned int，占4个字节。

以e3为例，最大的值是4294967296(gray)，这已经超出了 unsigned int 类型所能容纳的最大值，所以扩充为 long 类型，占8个字节。

e5相对于e4只是第1个变量手动赋值为了－1，但是它们的大小分别是8和4。这是为什么呢？前面我们提到过，枚举默认是int类型，当枚举内最大的值超过了int类型的话，编译器默认会自动扩充为 unsigned int 类型，但是e5内的成员由于存在负数，不能使用 unsigned int 类型，所以它只能扩充为 long 类型。

讲到这里，你应该已经明白了，如果枚举内最大的值超过了 signed long 类型的最大值，首先它会尝试使用 unsigned long 类型，如果可以的话那它的大小还是占8个字节；但是如果最大值超过了 unsigned long 类型，或者枚举内存在负数，那么它会继续扩充字节(扩充的字节也需要考虑对齐)。

一般情况下枚举都只占用1个int类型的大小，因为这已经能满足绝大部分的使用场景了。

补充

1. 为什么要进行内存对齐呢？

  1. 提高CPU访问内存的效率。
  2. 提高代码的可移植性，不是所有的硬件平台都能访问任意地址上的数据。

2. 为什么不统一外字节对齐，而提供#pragma pack(n)修改？

  1. 提高代码的在不同平台下的兼容性，例如可以使用 #pragma pack() 指定对齐字节，这样无论在32位环境下还是在64位环境下对齐字节数都一样的，可以避免因为对齐字节数不同导致的错误。
  2. 可以让开发人员更好的利用内存空间，让他们自由的选择优化空间还是优化时间；#pragma pack(n) 命令中的n越大的情况下CPU访问速度越快，内存占用空间越大；越小的情况下CPU访问速度变慢，内存占用空间越小。
  3. #pragma pack(n)命令中的n只能是1、2、4、8、16，如果是其他值的话，编译器会选择离这个值最近的最大值，例如 #pragma pack(3) 编译器会按照是4去对齐(我只在Mac OS系统上测试过，其他系统不确定是否符合。)。
